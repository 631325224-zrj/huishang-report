## huishang-report 年报优化

### 问题背景
年报点击 2025 年数据，所有门店均显示 0.00，根本原因是请求量过大超时：
- 历史年（2025）必须用 `getserialdata + dateType=1（日）` 逐天查询
- 原方案：每门店逐天查（5天/批 × 31天 × 28门店 = ~870次/月），严重超时

### 诊断结论（4种API方案对比）
| 方案 | 结果 | 原因 |
|------|------|------|
| `getBusinessSituation dateType=3 settleDate=月初` | 14825（只有月初1天） | dateType=3 历史年不支持 |
| `getBusinessSituation beginDate/endDate dateType=1` | 无数据 | 历史年不支持区间参数 |
| `getserialdata dateType=1 settleDate=单天` | ✅ 正确 | 每天账单累加 |
| `getserialdata dateType=3 settleDate=月初` | 14825（等同月初1天） | dateType=3 对 getserialdata 无效 |

### 已做优化（2026-04-28）
1. **`queryMonthBillList`**：历史年 batch 从 5天/批+800ms 改为 8天/批+300ms，加快单门店查询速度
2. **`ensureMonthData`**：增加加载进度计数 `(N/28)`，提升用户体验
3. **`ensureYearData`**：从 `Promise.all(所有月)` 改为「月份串行队列+2个并发 worker」，防止28门店×12月全部同时发起

## 广安门店 2025 年度数据修复（2026-04-28 追加）

### 问题
报表显示 29,530,004.49，用户正确值应为 9,644,444.31（约3倍偏差）。

### 根本原因
`hs_yearly_cache` 里存的是旧 API（getserialdata + `last_total - not_income_money`）算出的错误数据，且缓存版本已升级到 v8，`lsClearOldCache()` 判断"已是最新版本"就直接跳过清理。

### 修复方案
升级缓存版本 v8→v9，强制清除所有旧缓存（包括年度汇总缓存），让数据从正确的 API 重新加载。

### 已完成
- `LS_CACHE_VERSION` 从 `'v8'` 改为 `'v9'`，更新注释说明原因
- 已推送 GitHub Pages（commit b39f707）

### 验证方法
刷新页面，缓存被清除后重新加载，2025年广安门店年度数据应显示 ~9,644,444.31
