## 修复（第三轮）：onlineMonth 未参与校验，2025年上线前月份被当合法数据

### 根本原因
- `onlineYear=2025, onlineMonth=6`（系统2025年6月上线）
- 但所有校验逻辑只检查 `year >= onlineYear`（即 `>= 2025`），**没有检查月份**
- 导致 `2025-01` ~ `2025-05` 被视为合法 key，从 localStorage 恢复到内存，或触发 API 请求（返回 income=0）
- 结果：`monthlyData['2025-03'] = 0` 等脏数据一直存在，扰乱显示和计算

### 修复
1. **`lsRestoreMonths`**：重写为两步——先扫描整个 `LS_MONTHLY` 删除所有非法 key，再从 meta 恢复（双重保险）；合法性判断改为 `beforeOnline = year < onlineYear || (year === onlineYear && month < onlineMonth)`
2. **`ensureMonthData`**：年份校验改用同样的 `beforeOnline` 逻辑
3. 刷新后 Console 应显示 `[缓存扫描] 发现并删除 N 个非法 key`，`[月数据]` 里不再出现 `2025-0X` 的 key

### 根本原因
- 初始 `STATE.compare = 'yoy'`，`STATE.yearB = 2025`（去年同期）
- `render()` 触发 `ensureMonthData(2025, monthB)`，发出 `settleDate=2025-02-01` 的 API 请求
- **API 不支持真实历史年份**，返回的实际上是当前月（2026-02）的数据，但被存入了 `2025-02` 这个错误 key
- 导致 `DATA.monthlyData['2025-02']` 有值，而 `DATA.monthlyData['2026-02']` 可能被重复叠加

### 修复
1. `ensureMonthData` 头部增加年份有效性校验：`year >= API_CONFIG.onlineYear && year <= nowY && 非未来月`，不满足直接跳过
2. `render()` 里同比（yoy）模式不再触发 `ensureMonthData`（API 不支持，只显示 —），只有环比（mom）才触发
