# 2026-04-05 工作日志

## ensureMonthData 数据问题修复（两轮）

### 问题1：API超时导致数据加载失败
- 现象：页面显示"数据加载失败 Failed to fetch"
- 原因：API 连接超时，无超时控制
- 修复：添加 AbortController 10秒超时 + 重试一次机制

### 问题2：数据重复累加（二月显示过大）
- 现象：大蜀山店2月显示 692168.75（正确），但实际值偏大
- 原因：初始加载35天数据已包含2月部分数据，ensureMonthData 再加载完整2月时又重复累加
- 修复：ensureMonthData 加载前先清除该月原有的部分数据

### 问题3：清零后二月显示 0.00（本次修复）
- 现象：2月数据显示 0.00，Console 显示"数据已存在"跳过加载
- 根本原因：清零后 key 仍存在于 monthlyData，`key in DATA.monthlyData` 判断为 true 跳过加载
- 修复方案：
  1. 新增 `loadedMonths` Set，专门记录"已完整加载"的月份（与 key 是否存在分离）
  2. `ensureMonthData` 改为检查 `loadedMonths.has(key)` 而不是 `key in monthlyData`
  3. 初始加载完成后把当前月标记进 `loadedMonths`（当前月是实时数据，不需要重复加载）

### 核心教训：清零操作 + in 判断的陷阱
清零后 key 仍然存在，导致判断逻辑错误。应使用独立的状态标记，不要依赖数据值来判断是否已加载。

### 问题4：环比月份数据始终为 0（本轮修复）
- 现象：2026年2月在环比对比时显示 0.00，Console 显示"[加载完成] 2026-02: 0"
- 根本原因：初始 35 天窗口只包含 2 月的后半段（约 2 月 6-28 日）。当 ensureMonthData 加载完整 2 月时：
  1. 先清零该月旧数据
  2. 发起 28 天 × 28 门店 = 784 个 API 请求
  3. 2 月 1-5 日没有缓存，需要请求 API，但 API 大量超时，导致 income 全为 0
- 修复方案：ensureMonthData 重写，**优先从 SHOP_DATA_CACHE 读取已缓存的日期**，仅对未缓存日期发起真实 API 请求，大幅减少请求量。
- 缓存 key 格式确认：`${centerId}_${shopId}_${date}_day`（与 queryRevenue 完全一致）
