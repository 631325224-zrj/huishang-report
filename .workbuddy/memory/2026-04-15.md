# 2026-04-15 工作记录

## 年报表头闪烁修复

**问题**：年报点击后，表头先显示日期、再变月份、最后变年份，多次闪烁。

**根因**：
1. `getSummaryRevenue(isCurrent, shopCenterId, dim)` 接受了 `dim` 快照参数，但 else 分支（year 维度）内部仍使用 `STATE.dim` 做判断，导致 year 分支实际被 fallthrough 到错误维度。
2. `render()` 是 async 函数，`preloadPromises` 后台并发执行时，每次月份加载完毕后调用 `render()`，此时 `STATE.dim` 已变为 `year`，但 `getSummaryRevenue` 取到的是旧的 `day` 快照，表头被覆盖。

**修复**：
- `getSummaryRevenue` 改为：传入 `dim` 时直接用，不再回落到 `STATE.dim`；未传时（exportData 等外部调用）降级用 `STATE.dim`（year 分支）。
- 升级缓存版本 v5→v6（原版本同时有预加载双重累加问题，见下）

## 2025年数据偏大修复

**问题**：报表显示 2025 年朝阳门店 11,354,146，收银系统后台是 11,352,045，差约 2,100 元。

**根因**：预加载（初始化第三阶段，`preloadPromises`）是后台并发执行的，但没有检查"目标月份是否已经从 localStorage 缓存恢复"，导致：
- 第一步：`lsRestoreMonths` 从缓存恢复 2025-09~2025-11 数据（含 yearlyData 累加）
- 第三步：预加载又对同月份发一次 API 请求，把结果再累加一次

**修复**：
- `preloadPromises` 每次执行前检查 `loadedMonths.has(pk)`，已加载则跳过
- API 完成后再次检查（防并发竞争）
- 升级缓存版本 v6，清除含双重累加的脏缓存

## 出租率报表功能开发（同日追加）

**需求**：新增"出租率"维度，展示各门店包间午/晚市出租率。

**实现**：
1. `SHOP_ROOMS`：27家门店包间数量配置（从汇总表读取）
2. `queryReserve`：调用 `datatransfer/getReserveList`，按 mealType=1/2 分午晚市查询，过滤客位类型含"包"的预订，排除取消状态
3. `queryOccupancyAll`：并发查询所有门店，返回 `{shops: {centerId: {noon,evening,rooms}}}`
4. 出租率维度按钮：切换后隐藏普通报表区域（summary/charts/revenue-table），显示出租率专区
5. 出租率统计维度：日/月/年，点击"查询"按钮触发
6. 汇总卡片：午市/晚市/全天出租率 + 进度条（≥60%红色，30-60%橙色，<30%蓝色）
7. 门店明细表：各门店包间数、午预订数、晚预订数、各市出租率、全天出租率，按区域分组

**计算公式**：
- 午市出租率 = 午市包间预订数 / 总包间数
- 晚市出租率 = 晚市包间预订数 / 总包间数
- 全天出租率 = (午+晚合计) / (总包间数×2)

**注意**：getReserveList 接口需验证实际字段名（list/reserveList，seatTypeName/seat_type_name 等），如接口不存在则需找其他预订查询接口。

**问题**：年报表头显示日期格式，门店数据也是日报数据而非年报数据。

**根因**：`ensureYearData(onReady)` 的回调是 `() => render()`，`render()` 执行时重新读 `STATE.dim`。若加载期间初始化的其他异步任务（如预加载日数据）完成并触发了日报渲染，导致 `STATE.dim` 已变为 `day`，回调执行时拿到的是错误维度。

另外 `renderTable` 的 BLOCKED 保护逻辑漏写了 `return`（只打了 log 没有真正阻断），后来加上 return 又会误阻断用户切回日报的渲染。

**修复**：
1. `ensureYearData` 新增 `dimSnapshot` 参数，加载完成时校验 `STATE.dim === dimSnapshot` 才触发渲染
2. 移除 `renderTable` 里错误的 BLOCKED return 逻辑

## 2025年年报数据为0修复（第二轮，同日追加）

**问题**：年报点击后，2025年所有门店数据显示0.00。

**根因**：`queryMonthBillList` 使用 `getserialdata` 逐天逐页分页查询，对2025年历史数据每月需 31天×28门店×N页 = 千次以上请求，必然超时导致返回空数据。

**修复**：将 `queryMonthBillList` 改为使用 `getBusinessSituation`（`dateType=2` 月维度），每门店只需1次请求，直接返回整月 `income.incomeTotal`，与日数据口径完全一致。缓存版本升级 v6→v7，清除旧 getserialdata 缓存。

**教训**：历史年份月度查询不能用逐天明细API，必须用月维度汇总API（getBusinessSituation dateType=2）。
